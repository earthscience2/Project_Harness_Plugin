# 전역 영역 — 영역 하네스 설명서

## 이 하네스는 무엇인가
프로젝트 지식을 **영역(Area)** 단위로 쪼개 관리하는 SSOT 하네스다.
각 영역은 AI 한 세션이 한 번에 소화할 크기(≤ 1M 토큰)로 제한되며, 전역 영역(이 문서)에서 출발하는 그래프로 연결된다. 항상 이 문서부터 읽는다.

**코드는 영역 안으로 수용한다(contained).** 하네스는 디렉터리 구조 자체를 영역 트리와 정합시킨다. `/harness-adopt` 가 승인된 영역 트리에 맞춰 소스를 각 말단 영역의 `src/` 안으로 **물리 이동**(`git mv`)하고, 깨지는 import·실행/배포 경로를 전부 재작성한 뒤 빌드/테스트 통과를 게이트로 둔다. 레지스트리의 `sources` 는 이제 **영역 내부 경로**(`areas/<path>/src/...`)를 가리킨다. 빈 프로젝트면 새 코드를 처음부터 `areas/<path>/src/` 안에 만든다.

## 영역의 정의
- **전역 영역**: 마인드맵 루트. 설명서·구조·사용법(이 파일 하나).
- **중간다리 영역**: 연결만 하는 노드. `_area.md` 에 자식 목록만. 실내용 없음.
- **말단 영역**: 실내용이 있는 곳. 영역 폴더 = **`src/`(이 영역이 소유한 실소스, 이동해 들어옴) + `content/`(전용 자료) + `ssot/`(NOTES.md + ssot 문서)**. `NOTES.md` 는 `ssot/` 에 두고, 그 안의 소스 링크는 영역 내부(`src/...`)를 가리킨다.

## 영역 규칙
1. **코드는 영역으로 수용(contained).** adopt 시 소스를 `git mv` 로 `areas/<path>/src/` 안에 넣고, 깨지는 참조(import·실행/배포 경로)를 재작성한다. 디렉터리 구조 = 영역 트리. 한 모듈이 ≥2 영역에서 쓰이면 `areas/shared/src/` 로 모으고 거기서 import 한다(아래 「공유 코드」).
2. **경계는 의미로 가른다(1차).** 한 영역 = 하나의 책임 · 하나의 계약 · 한 명의 담당. 흐름이 자연스러우면(`백엔드 → API`) 그대로. 의미가 맞으면 작아도 합치지 말고, 갈리면 남아도 나눈다.
3. **토큰은 2차 가드레일.** 말단 하나가 한 세션(≤ 1,000,000)에 안 들어오면 분열. 크기는 손으로 적지 않고 `node check-registry.mjs` 가 추정.
4. **중간다리는 자식 ≥ 2 일 때만.** 단일 자식 중간다리·중첩을 위한 중첩 금지.
5. **실 내용은 말단에만.** 전역·중간다리는 길잡이.
6. **공유 ssot 는 정본 1파일을 여러 영역이 참조.** 정본 `_ssot/<id>.md`, `shared_ssot.shown_in` 명시.
7. **모든 ssot 문서는 frontmatter `category` 필수** — `규칙 | 강제 | 설명 | 기억`.
8. **변경하면 로그 한 줄.** 영역/ssot/일정/자동화를 바꾼 주체가 `logs/log.jsonl` 에 append.

## ssot 카테고리
| category | 뜻 | AI 가 다루는 법 |
|----------|----|----------------|
| **규칙** | 권장 규약·표준 | 기본 준수, 합당하면 예외 |
| **강제** | 반드시 지킬 제약 | 위반 불가. 어기면 중단·재설계 |
| **설명** | 참고 맥락 | 읽되 구속력 없음 |
| **기억** | 결정 이력 | 되돌리기 전 반드시 확인 |

## 영역 레지스트리 (단일 소스)
> 진실은 [`registry.json`](registry.json) 한 파일. 영역 추가/분열/공유 변경 시 그 파일만 고친다(이 문서에 트리 복제 안 함).
> 편집 후 점검: `node check-registry.mjs`. 사람용 트리는 마인드맵(`harness/index.html`)이 실시간으로 그린다.

## 공유 코드
한 모듈이 ≥2 영역에서 import 되면 한 영역에만 못 넣는다. `/harness-adopt` 는 import 그래프로 이런 모듈을 검출해 전용 **shared 말단 영역**(`areas/shared/src/`)으로 모으고, 소비 영역들이 거기서 import 하게 한다(영역 간 import 허용). 깔끔히 안 떨어지는 코드가 많으면 자동 이동하지 말고 분할안을 사용자에게 되묻는다.

## 사용법
1. **항상 전역부터** 읽고 `registry.json` 에서 작업할 말단을 찾는다.
2. 흐름대로 내려간다: 전역 → 중간다리 → 말단.
3. **편집은 말단에서만.** 실제 코드는 그 영역의 `src/` 안에서 고친다.
4. 공유 ssot 수정 = 정본 1파일 → 참조 영역 전부 반영(`shown_in` 확인).
5. 분열은 의미가 갈릴 때 먼저, 그리고 검사기가 예산 초과를 경고할 때. 이동(`move`/`split`)은 파일 `git mv` + 참조 재작성 + 빌드 게이트를 동반한다(`/harness-area`).

## 사이드바 (harness/index.html)
- **영역**: 마인드맵. **일정**: `schedule/`. **자동화**: `automation/`(외부 API 키 기반 스크립트). **Claude Code**: Claude Code 로 도는 자동화(루프·훅·예약). **로그**: `logs/log.jsonl`. **보고서**: `reports/`(HTML). **설정**: 키(localStorage; `serve.py` 면 `automation.env` 로 저장).
- `serve.py` 로 띄우면 「테스트/실행」 버튼이 스크립트 자동화를 실제 실행, 켜기/끄기를 파일에 바로 기록. `python -m http.server` 면 읽기 전용.
