기존 연구 자료를 연구 하네스로 **수용(contained)** 한다. 범위: $ARGUMENTS (없으면 저장소 전체).

> **관리 하네스 `/harness-adopt` 와 같은 안전 규칙**(승인 전 무이동 · clean tree · `git mv` 이력 보존 · 참조 재작성 · 게이트 · 실패 시 전체 롤백)을 따른다. **차이: 상위 골격이 잠겨 있다.** 새 **최상위** 영역은 못 만든다 — 기존 자료를 **고정 6영역**(전역 + 5: 가설/질문 · 데이터/자료 · 실험 · 인사이트 · 보고서) 중 하나로 분류한다. **단, 한 영역에 자료가 많아 의미가 갈리면 그 영역 *하위*에 세부 영역을 만드는 것은 허용**(예: `data` 밑에 `data/corpus`·`data/labels`). 상위 6만 고정, 분열은 그 아래에서만.
>
> **영역 폴더 = 정확히 두 칸**(관리 하네스와 동일): `ssot/`(NOTES.md 본문 + `<id>.md` ssot 문서) + `content/`(자료·코드 — 폴더 구조 그대로). 별도 `src/`·`data/` 층 없음. 자료는 `content/` 직속.

## 절차

1. **골격 보장**: 루트에 `registry.json` 이 없으면 먼저 `/harness-research-init` 절차를 수행. 이미 있는데 `profile` 이 `research` 가 아니면 **중단**(서비스 하네스다 → `/harness-adopt` 안내).

2. **자료 스캔**: 디렉터리·파일 유형을 파악(Glob/Grep/Read). 연구 자료를 5말단 기준으로 읽어낸다:
   - **가설/질문** — 리서치 질문·가설 노트·`status` 가 붙을 만한 명제(`*.md` 질문/가설).
   - **데이터/자료** — 데이터셋·코퍼스·시뮬 입력(`*.csv`/`*.parquet`/`*.json`/원천 자료).
   - **실험** — 실험·학습·평가 코드, 노트북, 셋업 스크립트(`*.py`/`*.ipynb`/`train*`/`eval*`).
   - **인사이트** — 누적 발견·결론 노트.
   - **보고서** — 완성 산출물(HTML/PDF/슬라이드/리포트).

3. **분류 (구조 제안 아님)**: 각 파일을 위 5말단 중 하나로 매핑한다. **어느 말단에도 깔끔히 안 떨어지는 파일은 추측 이동 금지** — 「보류」로 분리해 사용자에게 되묻는다.

4. **결정은 선택창으로** — 분류 초안을 보여준 뒤 `AskUserQuestion` 으로 다음을 한 번에 묻는다(추측하지 말고 사용자가 고르게):
   - **이동 방식**: `git mv`(이력 보존 물리 이동, 기본) / `cp`(원본 유지 복사 — 외부에서 경로 참조하는 데이터일 때) / 자료별 혼합.
   - **보류 파일 처리**: 「데이터/자료」로 기본 수용 / 이번엔 제외하고 보고만 / 하나씩 말단 지정.
   - **대용량 데이터**: `content/` 로 이동 / 원위치 두고 `ssot/NOTES.md` 에 경로만 링크(레포에 큰 바이너리 안 들이기).
   - **코드 게이트**: 실험 코드가 있으면 이동 후 빌드/테스트 실행해 통과 게이트 / 생략(코드 없거나 자료뿐).
   - **세부 분열**: 한 골격 영역에 자료가 많아 의미가 갈리면 그 **하위에 세부 영역**을 만들지(예: `data/corpus`·`data/labels`) / 한 영역에 평평하게 둘지. 상위 6은 못 늘리지만 하위 분열은 허용 — 만들면 그 골격 영역은 말단→중간다리로 승격.

5. **사용자 확인**: 최종 분류표(파일 → `areas/<말단>/content/...`)와 4번 선택 결과를 요약해 보여주고 **승인받는다**. 큰 변경이므로 이동 전에 멈춘다.

6. **선결 조건 강제**: `git mv` 를 고른 경우 `git status --porcelain` 으로 작업트리가 **clean** 인지 확인. 더러우면 **중단**(커밋/스태시 안내). git 저장소 아니면 `git mv` 불가 → `cp` 로 안내하거나 중단.

7. **이동**: 선택한 방식으로 각 파일을 `areas/<말단>/content/` 로(폴더 구조 그대로). `git mv` 는 이력 보존, `cp` 는 원본 유지. 대상 폴더 먼저 생성. 대용량+링크 선택분은 옮기지 말고 NOTES 링크만.

8. **참조 재작성 (이동분만)** — 이동으로 깨지는 참조를 고친다:
   - **노트북/스크립트**: 데이터 상대경로(`pd.read_csv("data/...")`), import, 설정 파일 경로.
   - **실행/배포 문자열**이 있으면 `/harness-adopt` 와 동일하게 처리.
   - **자동 재작성 불가능한 참조는 이동하지 말고 목록으로 보고**(추측 수정 금지).

9. **게이트**(4번에서 켠 경우만): 실험 코드의 빌드/테스트를 **실제 실행**(`pytest`, `npm test` 등). 자료만 옮겼으면 생략.
   - **실패 시 전체 롤백**: `git restore --staged . && git restore . && git clean -fd`(또는 `git reset --hard HEAD`). `cp` 였으면 복사본 삭제. 원인 보고 후 중단.

10. **하네스 작성**(게이트 통과 후):
    - 각 말단 `ssot/NOTES.md` 본문 갱신(코드/자료 링크는 `../content/...` 상대경로). 필요하면 `ssot/<id>.md`(frontmatter `category` 필수 — 인사이트는 `category: 기억`).
    - `registry.json` 갱신: 자료가 들어간 말단의 `ssot` 배열에 새 `<id>` 등록. **세부 분열**을 골랐으면 그 골격 영역을 `bridge` 로 승격(`children` 채우고 `ssot/NOTES.md` 를 자식 목록으로 전환, `content/` 는 비움; `path` 는 유지)하고 하위 말단을 추가(부모는 그 골격 영역, `path` 는 그 아래 `areas/<골격>/<자식>/`). **`skeleton`·`locked`·root 직속 6영역은 절대 건드리지 않는다 — 새 영역은 골격 영역 하위로만.**

11. `node check-registry.mjs` → 오류 0 까지. **잠금 검사 통과 필수** — 새 *최상위* 영역을 만들었거나 골격 영역을 재배치했으면 실패하므로 되돌린다(하위 분열은 통과). `content/` 빈 말단·NOTES 누락 경고 확인.

12. `logs/log.jsonl` 에 append: 이동별 `{"op":"code:move","target":"<말단id>","from":"<원경로>","to":"areas/<말단>/content/...","summary":"..."}` + 마무리 `{"op":"research:adopt","target":"global","summary":"N개 자료 수용"}`.

13. 결과 요약(말단별 수용 파일 수 · 보류/링크 처리 · 게이트 결과) + 마인드맵 `harness/` · 보고서 `reports/index.html` 안내.

## 규칙
- **구조 불변.** 영역을 추가/삭제/분열하지 마라 — 잠긴 6영역 그대로, `content/` 만 채운다. 변형은 영역이 아니라 `runs` 의 `exp` 태그.
- 승인 전 이동·수정 금지. `git mv` 는 clean tree 위에서만, 실패하면 전체 롤백.
- 결정 사항(이동 방식·보류·대용량·게이트)은 추측하지 말고 `AskUserQuestion` 선택창으로 사용자에게.
- 자동 재작성 불가능한 참조는 추측 말고 보고. 게이트를 켰는데 통과 못 하면 수용을 확정하지 않는다.
